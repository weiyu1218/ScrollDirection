# 登录项注册修复方案

## 目标

修复 ScrollDirection 首次启用登录启动时，`SMAppService.mainApp.status` 返回 `notFound` 导致注册逻辑提前终止的问题。保持滚动处理、权限处理和现有登录项状态行为不变。

## 调用链路

应用启动路径：

```text
ScrollDirectionApp.init
  -> AppState.startAtLaunch
  -> AppState.applyDesiredState
  -> AppState.registerDefaultLoginItemIfNeeded
  -> SystemLoginItemController.setEnabled(true)
  -> SMAppService.mainApp.status
  -> SMAppService.mainApp.register()
```

菜单开关路径：

```text
MenuBarView 登录时启动 Toggle
  -> AppState.setLoginItemEnabled
  -> SystemLoginItemController.setEnabled
  -> SMAppService.mainApp.register() 或 unregister()
  -> AppState 刷新 loginItemStatus
```

错误由 `SMAppService` 抛出，经过 `AppState.setLoginItemEnabled` 转换为菜单中的真实错误信息。该调用链不进入滚动事件回调，不涉及并发、缓存、网络、数据库或事务。

## 诊断证据

- `/Applications/ScrollDirection.app` 的 bundle identifier 为 `com.weiyu1218.ScrollDirection`，签名有效，安装位置固定。
- 应用进程内 `SMAppService.mainApp.status.rawValue` 为 `3`，对应 `.notFound`。
- 现有 `SystemLoginItemController.setEnabled(true)` 在 `.notFound` 分支直接抛出 `LoginItemControllerError.notFound`，没有调用 `register()`。
- 在同一应用进程中直接执行 `try SMAppService.mainApp.register()` 成功。
- 注册后状态变为 `.enabled`，`sfltool dumpbtm` 显示 `ScrollDirection`、`com.weiyu1218.ScrollDirection`、`/Applications/ScrollDirection.app`，Disposition 为 enabled。
- Apple SDK 将 `register()` 定义为注册服务并由系统返回真实失败原因，因此注册调用应当作为能否启用的最终判定。

## 根因

控制器错误地将查询状态 `.notFound` 当成不可恢复终态。真实系统数据表明，在当前主应用登录项场景中，`.notFound` 后直接注册可以成功。因此前置状态判断阻止了有效注册，并用本地推断错误覆盖了系统真实结果。

## 修改方案

在 `SystemLoginItemController` 内建立单一的登录项操作决策：

```text
启用 + enabled             -> 无操作
启用 + requiresApproval    -> 返回需要用户批准错误
启用 + notRegistered       -> register
启用 + notFound            -> register
禁用 + enabled             -> unregister
禁用 + requiresApproval    -> unregister
禁用 + notRegistered       -> 无操作
禁用 + notFound            -> 无操作
```

以内部纯值 `LoginItemOperation` 表达 `none/register/unregister`，让所有状态组合可被单元测试覆盖。`register()` 或 `unregister()` 的系统错误原样向上抛出。删除不再使用的本地 `notFound` 错误。

## 文件影响

- `ScrollDirection/LoginItemController.swift`：修改状态决策和执行逻辑。
- `ScrollDirectionTests/LoginItemControllerTests.swift`：覆盖八种启用状态组合。
- 其他源文件不修改。

## 测试与验收

1. 先添加 `.notFound + 启用 -> register` 等操作决策测试，确认当前实现不满足测试契约。
2. 实现最小决策逻辑并运行登录项测试。
3. 运行全部 Swift 测试和 Debug 构建。
4. 重新安装后验证登录项状态为 `.enabled`，并在系统登录项数据库中存在正确 bundle identifier 和固定路径。
5. 执行代码差异审查后提交独立 `fix:` 中文提交。

## KISS 自检

1. 这是现实问题：真实系统返回 `.notFound`，现有代码未注册，登录项数据库无记录。
2. 更简单的做法：仅修正状态到注册操作的映射，不改 AppState 或添加辅助应用。
3. 不破坏现有功能：滚动链路完全不变；已经启用、需要批准和注销路径保持原语义。
4. 当前项目需要：登录时启动是已批准并已实现的明确功能，当前验收未通过。

## 性能与风险

- 性能：仅在启动首次注册或用户切换登录项时执行一次状态判断，没有事件热路径开销。
- 风险：若 `.notFound` 源于不可注册的安装或签名问题，`register()` 会返回真实系统错误并由现有界面展示，不会伪装成功。
- 回滚：单个独立提交仅涉及登录项控制器和测试，可在不影响滚动功能的情况下回退。
